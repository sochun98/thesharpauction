## 2026-04-19 (7)

**Q. 좌표보완 후 지도조회 시 GUI가 멈추는 원인은?**

A. `_on_load_map` → `_render_map` 전체가 메인(UI) 스레드에서 실행되어 블로킹 발생. 구체적으로 ①`load_map_data()` DB 조회 ②folium CircleMarker 수천 개 생성 ③`m.save()` HTML 저장 ④HTML 파일 읽기+스크립트 삽입이 모두 UI 스레드에서 실행됨. 좌표보완 후 좌표 보유 건수가 늘수록 렌더링 부하가 커져 더 오래 멈춤.

수정: `MapLoadThread(QThread)` 추가 — DB 조회+folium 렌더링+HTML 저장을 백그라운드에서 수행. 완료 시 `done_signal(html_path, rows)` emit → 메인 스레드에서 `web_view.load()`와 테이블 채우기만 처리. `_render_map()` 메서드 제거.

## 2026-04-19 (3)

**Q. 지오코딩 실패는 왜 발생하는가?**

A. 로그 분석 결과 3가지 원인:
1. **층/호수 포함 주소** (가장 많음): 네이버 Geocoding API는 도로명 주소까지만 인식. "2층203호", "지하1층비112호" 등이 포함되면 매칭 실패.
2. **미등록 신개발사업지 명칭**: "식사2도시개발구역5블록3로트", "고양장항지구에이-5블록" 등 사업구역명은 도로명주소 DB에 없음.
3. **기타 소수**: "둔촌동 둔촌동" 처럼 동명 중복 저장된 DB 오류, 쉼표 이후 층 정보, 지번주소 미등록.

개선안: `_clean_address`를 강화해 층/호 제거 → 쉼표 이후 제거 등 단계적 fallback 재시도.

## 2026-04-19

**Q. 지도 탭 법원경매상세보기에서 사건번호 "2025타경100874"를 "10087"로 검색해서 사건을 못 찾는다. 왜 그런가?**

A. 법원경매 사이트의 WebSquare 기반 입력 필드(`ibx_auctnCsSrchCsNo`)에 **maxlength=5** 제한이 있어서 발생하는 버그.

기존 코드는 `case_no_field.fill("타경100874")`처럼 입력했는데, 이 필드는 숫자만 처리하고 maxlength로 자르기 때문에 `"100874"` → `"10087"` (앞 5자리) 로 절단됨. 4자리 사건번호("1345")는 5자리 제한 이하라서 정상 동작했음.

**수정:** `detail_scraper.py`에 `_fill_case_no()` 헬퍼 추가 — JS로 `el.maxLength = -1` 해제 후 `.fill()` 호출. 입력 순서도 `parsed["num"]` (숫자만) 먼저 시도, 실패 시 `"타경" + num` 순으로 변경. `auction_gui.py`의 `OpenBrowserThread`도 동일하게 수정.

## 2026-04-18 (3)

**Q. GUI 지도 탭에서 2025타경1345의 상태가 "낙찰"로 표시되는데, 법원경매사이트 상세보기에서는 "최근입찰결과: 2026.04.14 유찰"로 표시된다. 왜 다르게 표시되는가?**

A. 두 값이 서로 다른 시점·기준의 상태를 보여주기 때문이다.

| 구분 | 출처 | 의미 |
|------|------|------|
| GUI "낙찰" | API `mulStatcd = "04"` (수집 시점 스냅샷) | 사건 전체의 처리 상태 (과거 낙찰 이력 반영) |
| "2026.04.14 유찰" | 법원경매사이트 실시간 조회 | 가장 최근 매각기일의 개별 결과 |

**가장 가능성 높은 시나리오**:
1. 이전 매각기일에 낙찰 → `mulStatcd = "04"` 로 기록됨
2. 매수인이 대금 미납하거나 법원이 낙찰허가를 취소
3. 재매각 절차 진행 — 그러나 GUI 수집 당시 API는 아직 `"04"` 상태를 반환
4. 2026.04.14 재매각 매각기일에서 입찰자 없어 유찰

**결론**: API의 `mulStatcd`는 수집 시점의 스냅샷이며 실시간 반영이 보장되지 않는다. DB의 상태는 참고 수준으로만 사용하고, 정확한 현재 상태는 법원경매사이트 상세보기로 직접 확인해야 한다.

## 2026-04-18 (2)

**Q. 시군구 단위 분할 수집, 상한 증가, GUI(수집+지도), MySQL 저장을 구현하라.**

A. 세 파일로 구현.

**src/db.py** (신규)
- MySQL `auction_cases` 테이블 (35개 컬럼 + lat/lng + 인덱스)
- `upsert_records`: ON DUPLICATE KEY UPDATE (case_no + item_no 기준)
- `load_map_data`, `load_case_detail`, `update_geocode`
- `load_ungeocode_records`: 좌표 없는 레코드 배치 조회

**src/collector.py** (수정)
- `collect_sido_year`: sgg_code 파라미터 추가, 상한 1000페이지, save_fn 콜백
- `collect_by_sigungu`: 시도 내 시군구 목록 × 연도 분할 수집
- `get_sigungu_list(sido_name)`: 시도별 시군구 목록 반환
- CLI: `--sigungu`, `--mysql` 옵션 추가

**src/collector_app.py** (신규) — `streamlit run src/collector_app.py`
- [수집] 탭: 시도/시군구/연도 선택 → collect_by_sigungu → MySQL upsert → 실시간 로그
- [지도] 탭: DB 데이터 조회 → folium 지도 (상태별 색상: 진행중=빨강, 낙찰=파랑, 재매각=주황)
             → 좌표 보완 버튼 (네이버 API 배치 지오코딩)
             → 사건번호 선택 → 상세보기 (Playwright 스크린샷)

**MySQL 환경변수** (`env_thesharpcharm.env`에 추가):
```
MYSQL_HOST=localhost
MYSQL_PORT=3306
MYSQL_USER=root
MYSQL_PASSWORD=(비밀번호)
MYSQL_DATABASE=auction_db
```

## 2026-04-18 (1)

**Q. 경기도 2026년 수집 결과는 어떻게 되었는가?**

A. 백그라운드 수집 완료. 결과:

| 항목 | 수치 |
|------|------|
| API 응답 총건수 | 34,132건 |
| 중복 제거 후 고유 레코드 | **6,968건** |
| 처리 페이지 | 500페이지 (상한 도달) |
| 소요 시간 | 약 10분 |

중복이 많은 이유: 한 사건에 여러 물건번호(동, 호수)가 있으면 API에서 각각 별도 행으로 반환됨. 34,132 → 6,968로 약 80% 중복.

**문제점**: 500페이지 상한 도달로 수집이 완전하지 않을 수 있음.
→ 시군구 단위로 분할 수집하는 방식으로 개선 필요.

## 2026-04-17 (1)

**Q. 오늘까지 만들어진 경매 사건번호를 수집할 수 있는가?**

A. 가능 여부는 단계에 따라 다릅니다.

**✅ 가능: 경매 기일이 지정된 사건번호 전수 수집**

API로 지역 + 기간 조합으로 수집할 수 있습니다.
- `statNum="05"` (전체) + `bidBgngYmd` / `bidEndYmd` 필드로 연도 필터 가능
- 수원 영통구 기준 2026년 경매 기일 있는 사건 → 약 240건 확인됨

**❌ 불가능: 경매 기일 없는 순수 접수 단계 사건번호**

법원이 접수하면 사건번호를 부여하지만, 경매 기일이 지정되기 전까지는 공개 API에 노출되지 않습니다.

실제 흐름:
1. 접수 → 사건번호 부여 (2026타경XXXX)
2. 법원 조사·감정 (수개월)
3. 경매 기일 지정 ← **이 시점부터 API에 노출**
4. 공개 입찰

**현실적으로 구축 가능한 것**

특정 법원 + 연도 기준으로 기일 지정된 사건을 전수 수집하는 기능.
예상 소요 시간:
- 수원지방법원 + 2026년: ~1분
- 경기도 전체 + 2026년: ~10분
- 전국 + 2026년: ~2시간
